# План разработки агента rza_rag

## Цель

Превратить текущий RAG-чат в детерминированный пайплайн генерации
Пояснительной Записки (ПЗ) стадии ОТР из сырого ТЗ на проектирование.

Принцип: **линейно-циклический конвейер, без агентской свободы выбора.**
Human-in-the-Loop на ключевых развилках.

---

## Этап 1: Компонент «RZA_Param_Extractor»

**Назначение:** извлечь из сырого ТЗ жёсткие маркеры объекта.

**Вход:** текст ТЗ (любой длины, из файла или буфера).

**Выход (JSON):**
```json
{
  "object_name": "ПС 110 кВ Стойленская",
  "voltage_hv": "110 кВ",
  "scheme_hv": "4Н",
  "transformers": [
    {"power": "25 МВА", "qty": 2}
  ],
  "scheme_lv": "2 секции с СВ",
  "technology": "цифровая ПС МЭК 61850",
  "special_nodes": ["БСК"],
  "raw_text_snippet": "..."
}
```

**Алгоритм:**
1. Получить текст ТЗ.
2. Отправить в LLM с промптом-экстрактором (строгий формат, перечень полей).
3. Распарсить JSON из ответа LLM.
4. Валидировать: все ли поля заполнены, нет ли выдуманных значений.
5. 🛑 **Точка контроля №1** — вывести карточку инженеру, ждать подтверждения/правок.

**Файл:** `pipeline/param_extractor.py`

---

## Этап 2: Компонент «RZA_Plan_Builder»

**Назначение:** построить оглавление будущей ПЗ на основе маркеров и шаблонов из базы.

**Вход:** утверждённые маркеры из Этапа 1 + коллекция `projects`.

**Выход (JSON-массив разделов):**
```json
[
  {"num": "1", "title": "Общие сведения", "sources": ["project_123.docx"]},
  {"num": "2", "title": "Защита трансформатора 25 МВА", "sources": ["gost_xxx", "project_456.docx"]},
  {"num": "3", "title": "Защита БСК", "sources": []}
]
```

**Алгоритм:**
1. Поиск по маркерам в коллекции `projects` → найти аналогичные ПЗ.
2. Извлечь структуры (оглавления) из найденных проектов.
3. LLM сопоставляет требования ТЗ со структурами → собирает итоговый план.
4. Если спецузел (БСК) не покрыт шаблонами — принудительно добавить раздел.
5. Под каждым пунктом — список источников (файлы проектов, СТО).
6. 🛑 **Точка контроля №2** — вывести план инженеру, ждать «План утверждён».

**Файл:** `pipeline/plan_builder.py`

---

## Этап 3: Компонент «RZA_Section_Writer»

**Назначение:** сгенерировать текст одного раздела ПЗ, строго по базе знаний.

**Вход:** пункт плана + маркеры объекта + доступные коллекции.

**Выход:** текст раздела (строкой) + список использованных источников.

**Алгоритм:**
1. Сформировать узкий поисковый запрос: маркеры схемы + тема раздела.
2. `hybrid_search` по коллекции `gost` (нормативка) + `manuals` (оборудование).
3. LLM генерирует текст раздела из найденных чанков.
4. **Жёсткие правила:**
   - Стиль: ГОСТ Р 2.105-2019, академический.
   - Только факты из чанков.
   - Если по узлу (БСК) ничего не найдено → **стандартная заглушка**:
     *«Параметры и состав защит определяются на этапе РД по согласованию с заводом-изготовителем».*
   - Выдумывать запрещено.
5. Вернуть текст + источники.

**Файл:** `pipeline/section_writer.py`

---

## Этап 4: Оркестратор пайплайна

**Назначение:** связать все компоненты в единый поток с контрольными точками.

**Алгоритм:**
1. Загрузка ТЗ → `Param_Extractor` → 🛑 точка контроля №1.
2. Утверждённые маркеры → `Plan_Builder` → 🛑 точка контроля №2.
3. Цикл по разделам плана:
   - `Section_Writer` для текущего раздела.
   - Потоковый вывод в UI (Streamlit/Langflow).
   - Кнопка Stop на любой итерации.
4. По завершении цикла — сборка всех разделов в финальный документ.
5. Экспорт в `.docx`.

**Интерфейс:** Streamlit (новый экран «Генерация ПЗ») или Langflow Canvas.

**Файл:** `pipeline/orchestrator.py`

---

## Этап 5: Сборка в .docx

**Назначение:** склеить все разделы согласно плану в документ.

**Алгоритм:**
1. Шаблон документа (титульный лист, рамки, штампы по ГОСТ).
2. Вставка сгенерированных разделов в порядке плана.
3. Автонумерация разделов, таблиц, рисунков.
4. Экспорт в `.docx`.

**Файл:** `pipeline/docx_assembler.py`

---

## Этап 6: Тестирование на реальных ТЗ

1. Подготовить 2-3 тестовых ТЗ (разные схемы, классы напряжения).
2. Прогнать полный пайплайн.
3. Сравнить результат с эталонными ПЗ.
4. Замерить точность (совпадение структуры, отсутствие галлюцинаций).
5. Итеративно править промпты.

## Приоритет реализации

| Приоритет | Компонент              | Почему                                      |
|-----------|------------------------|---------------------------------------------|
| 🔴 P0     | `Param_Extractor`      | Входной гейт, без него ничего не работает   |
| 🔴 P0     | `Plan_Builder`         | Определяет структуру всего документа        |
| 🟡 P1     | `Section_Writer`       | Циклическая генерация — ядро системы        |
| 🟡 P1     | `Orchestrator`         | Связывает всё воедино, интерфейс            |
| 🟢 P2     | `docx_assembler`       | Экспорт — финальный продукт                 |
| 🟢 P2     | Тестирование           | Валидация на реальных данных                |
